# Fugue CLI 改进验收标准

## 1. 目的

本文定义 Fugue CLI 改进完成后的验收标准。验收以“是否能只靠 CLI 完成复现、定位、归因、取证”为准，不以某个具体实现方案为准。

## 2. 总体验收条件

满足以下条件时，视为 Fugue CLI 达到可用标准：

- 一名只持有高权限访问凭证的工程师，不依赖 SSH、`kubectl`、浏览器开发者工具或额外脚本，仅使用 Fugue CLI，即可完成从公开 API 用户流复现到后端故障归因的完整调查。
- 对于“公开 API 报错或超时，但后端副作用已经提交”的问题，CLI 必须能明确回答三件事：
  1. 用户请求是否真的失败。
  2. 后端副作用是否已经发生。
  3. 真正失败发生在边缘层、网关层、控制平面层、工作负载层，还是底层 Kubernetes/exec 层。
- CLI 必须能输出一份可共享、可脱敏、可归档的调查证据包，使接手者无需再次现场登录环境即可继续分析。

## 3. 核心验收标准

### 3.1 公开 API 用户流复现

- CLI 必须支持声明式执行多步 HTTP 工作流，而不是只能调用控制平面原始 API。
- 工作流必须支持以下能力：
  - 多个 base URL。
  - JSON、表单、multipart 请求体。
  - Bearer Token、Cookie、Header 透传与变量替换。
  - 从上一步响应中提取字段、Cookie 或 Header，供后续步骤复用。
  - 为每一步记录请求摘要、响应摘要、耗时、状态码和关键响应头。
- 工作流必须能够覆盖典型用户流：账号注册、会话建立、凭证签发、资源创建、资源查询、资源删除。
- 某一步失败时，CLI 必须清楚指出失败步骤、失败类型、失败前已成功执行的步骤，以及是否建议继续做状态核验。
- CLI 必须支持 `--json` 输出，并提供稳定字段，便于自动化系统消费。

### 3.2 多平面连通性与路由诊断

- CLI 必须支持显式选择目标平面或目标入口，包括：
  - 公网入口。
  - 控制平面 API。
  - 指定控制平面实例。
  - 指定工作负载或租户作用域。
- 当同一资源在不同入口上结果不一致时，CLI 必须给出差异对比，而不是只返回单点错误。
- 对于 `502`、`503`、`504`、`403` 这类常见边缘层错误，CLI 必须额外输出：
  - DNS/TLS/HTTP 连接阶段是否成功。
  - 是否命中 CDN/边缘代理。
  - 关键响应头和诊断标识。
  - 是否可以被判定为边缘层问题，而不是后端业务错误。
- CLI 必须支持在多个已知控制平面之间做健康探测和对比，并明确指出问题是“某个入口不通”还是“后端对象本身不存在/失败”。

### 3.3 控制平面与工作负载关联诊断

- CLI 必须支持从任意一个入口标识反查关联对象，至少包括：
  - 请求标识。
  - 资源标识。
  - 操作标识。
  - 工作负载标识。
  - Pod 和容器标识。
- CLI 必须能生成一条时间线，按时间顺序展示：
  - 公开 API 请求。
  - 控制平面对象变化。
  - 操作状态变化。
  - 工作负载创建、ready、重启、删除。
  - 关键错误日志。
- 对于应用层异常，CLI 必须能直接拉取并展示对应服务的堆栈信息或错误日志，而不是要求操作者再去登录机器。
- CLI 输出必须能明确区分：
  - 资源未创建。
  - 资源已创建但尚未 ready。
  - 资源已 ready，但后续同步步骤失败。
  - 资源已删除或正在删除。

### 3.4 工作区、文件系统与 exec 诊断

- CLI 必须提供面向“文件系统/工作区访问失败”的专用诊断命令。
- 该命令至少要输出：
  - 目标资源当前 phase。
  - 目标 Pod 列表。
  - 被选中的 Pod。
  - 被选中的容器名。
  - 所有容器的存在性、ready 状态、重启次数和最近状态原因。
  - 目标路径、挂载根路径和访问模式。
- 当底层是 Kubernetes exec 失败时，CLI 必须将错误归类为明确的故障类型，而不是只返回一行原始报错。至少要区分：
  - Pod 不存在。
  - 容器不存在。
  - 容器未 ready。
  - exec stream upgrade 失败。
  - API Server 临时不可用。
  - EOF/连接中断。
  - 权限问题。
  - 路径不存在。
- CLI 必须支持自动补充上下文证据，包括：
  - Pod/container 状态快照。
  - 最近事件。
  - 相关日志片段。
  - 最近一次失败的原始错误文本。
- 在“容器名存在但尚未 ready”与“容器名根本不存在”这两类问题上，CLI 输出必须明显不同，不能混为一谈。

### 3.5 日志、堆栈与证据包

- CLI 必须支持一条命令拉取一组相关日志，而不是要求用户逐个服务手动查。
- 日志拉取至少应覆盖：
  - 入口服务日志。
  - 控制平面服务日志。
  - 目标工作负载日志。
  - 相关操作日志。
- CLI 必须支持按时间窗口、资源标识、请求标识过滤日志。
- 证据包功能必须满足以下要求：
  - 一条命令导出。
  - 自动脱敏。
  - 包含请求摘要、响应摘要、对象快照、时间线、日志片段和错误分类。
  - 支持单文件归档输出。
  - 支持 `--json` 清单，描述包内内容。

### 3.6 机器可读输出与自动化友好性

- 所有诊断命令都必须支持 `--json`。
- `--json` 输出必须有稳定 schema，并在版本演进时保留兼容策略。
- CLI 必须定义清晰的退出码语义，至少区分：
  - 成功。
  - 用户输入错误。
  - 权限不足。
  - 目标不存在。
  - 确认存在系统故障。
  - 无法判定，需要人工介入。
- CLI 必须支持 `--output-file` 或等价能力，将原始结果写入文件。
- CLI 必须支持 `--redact`，并默认开启敏感信息脱敏。

### 3.7 权限与安全

- 使用高权限访问凭证时，CLI 默认输出必须脱敏以下信息：
  - Bearer Token。
  - Cookie。
  - API Key。
  - 租户级密钥。
  - 响应中出现的秘密字段。
- CLI 必须允许显式请求“导出未脱敏原文”，但必须要求用户主动确认，并在输出中添加危险标记。
- 证据包和终端输出必须保证相同的脱敏规则。

### 3.8 性能与稳定性

- 针对 10 步以内的公开 API 工作流复现，CLI 在正常网络条件下应在 5 分钟内完成。
- 针对单个资源的关联诊断，CLI 应在 30 秒内返回首屏摘要。
- 针对单个资源的完整证据包导出，CLI 应在 2 分钟内完成，除非用户显式要求更大时间窗口。
- 当后端部分接口不稳定时，CLI 必须能保留已获取证据，并在最终结果中区分“已确认事实”和“未取到的证据”。

## 4. 场景化验收

### 4.1 公开创建请求报错，但副作用已提交

- 输入一次公开创建请求复现工作流后，CLI 必须能自动完成后置核验。
- 当公开接口返回错误或超时时，CLI 必须继续检查后端对象、操作记录和工作负载状态。
- 最终输出必须明确写出：
  - 外部接口是否报错。
  - 后端对象是否已经创建。
  - 哪个后续阶段失败。
  - 是否存在清理动作或重试建议。

### 4.2 公网入口异常，但直连控制平面正常

- 当公网入口失败而控制平面 API 正常时，CLI 必须将问题归类为入口/边缘层问题。
- 输出中必须包含两个入口的对比证据，而不是只描述单边失败。

### 4.3 文件系统访问失败来自底层 exec 问题

- 当文件系统 API 最终失败根因是底层 exec 问题时，CLI 必须显示完整故障链：
  - 资源状态。
  - Pod 选择结果。
  - 容器选择结果。
  - 容器状态。
  - exec 原始错误。
  - 归类后的故障类型。
- 诊断完成后，操作者无需再登录集群查看容器状态，即可完成初步归因。

### 4.4 需要应用层堆栈才能定位

- 当根因发生在某个服务的业务逻辑中时，CLI 必须能直接拉到对应时间窗口内的堆栈或错误日志。
- 验收标准是：操作者不需要 SSH 到机器，也不需要单独运行日志平台查询，就能看到足够定位的异常上下文。

### 4.5 原始 API 域名或入口不稳定

- 当某个默认 API 入口不可用时，CLI 必须能：
  - 明确报出入口级不可用，而不是泛化为资源错误。
  - 支持切换到替代入口或指定控制平面继续调查。
  - 在最终摘要里说明“调查结果依赖于哪一个入口”。

## 5. 完成定义

只有当以下条件全部满足时，才能认为 Fugue CLI 的这轮改进完成：

- 上述所有核心验收标准都有对应命令或等价工作流支持。
- 所有场景化验收均可在无 SSH、无 `kubectl`、无浏览器开发者工具的条件下通过。
- `--json` 输出、脱敏规则、退出码语义和证据包格式已经稳定并文档化。
- 一名不熟悉底层集群结构的工程师，拿到访问凭证后，能按文档独立完成一次端到端调查。

## 6. 非约束性命令形态建议

本节仅用于帮助收敛交互方式，不作为强制验收项。

- `fugue workflow run <workflow-file>`
- `fugue diagnose request <request-or-trace-id>`
- `fugue diagnose resource <resource-id>`
- `fugue diagnose fs <resource-id> --path <path>`
- `fugue logs collect <resource-id> --since <duration>`
- `fugue debug bundle <resource-id> --since <duration> --output <file>`

最终是否采用这些命令名不重要，重要的是能力和验收结果必须满足本文要求。
